# 症状

工作台「🎲 随机重逢」卡片右上角的换一篇按钮(↻),手机上点一下正着转过去,再点一下会反着转回来,而不是每次都往前转一整圈。

# 原因(两层,第一版只修了第一层)

**第一层**:`.obsidian/snippets/workbench.css`(本地文件,不入库,远程会话看不到)大概率把旋转挂在 `:hover` 伪类上:悬停转到 180°,移出悬停转回 0°。桌面端有真正的 hover/leave 事件,手机没有——点一下相当于"进入 hover"、再点一下相当于"离开 hover",两次点击在两个固定角度间来回摆动。

**第二层(第一版改完实测仍然不对才发现)**:改成 `rerollGlyph.animate([...])` 之后问题依旧,原因是 `renderReunion(container)` 会在同一次点击里**同步**清空并重建整张卡片(含按钮和 glyph 本身)。`.animate()` 触发的动画播放需要浏览器至少画出一帧,但代码紧接着就把承载动画的 DOM 节点整个销毁重建了——动画请求发出后还没来得及被看见,元素就没了,视觉上等于没转。这是"触发动画 + 立刻销毁触发动画的节点"这类写法的通病,不止这一个按钮会踩。

# 解法

1. 旋转本身:用 Web Animations API 播 0°→360°,和 hover/active 状态完全脱钩
2. **关键的第二步**:点击回调改成 `async`,先 `await anim.finished` 等动画真正播完,再调用 `renderReunion(container)` 去清空重建卡片——顺序上"先看完转一圈,再看到换了新的一篇",而不是两件事同时发生导致动画被吞掉。点击时顺手 `reroll.disabled = true`,防止动画播放期间再次点击并发触发。

已在 工作台.md 的 `reroll.onclick` 里改完(见 2026-07-06 二次修复的注释),不用碰本地 CSS。

以后工作台里任何"点击触发一次性动效,紧接着又要重建/替换承载动效的那块 DOM"的需求,都要记得让"重建"等动效播完再执行(`await anim.finished` 或等价的 `setTimeout`/`animationend`),不能在同一个同步流程里又播动画又销毁节点。
