# 决定

`CLAUDE.md` 是每次会话都无条件加载的文件,用户明确要求尽量精简。以后判断"这部分内容要不要留在 CLAUDE.md 里"时,优先级顺序是:

1. **先问"这条内容是不是只有特定任务才需要"**——如果只在处理某类具体任务时才用得上(比如工作台生成、Memory GC、notes 复核这类由特定触发词或特定自动化调用的流程),搬去独立文件(仿照仓库已有的 `RESOLVER.md` 先例),CLAUDE.md 只留一句指针
2. **只有确认内容是"几乎每次会话都可能用上"的通用规则时,才留在 CLAUDE.md 正文**,且尽量写成可执行的短句,不铺陈历史narrative(何时改的、用户哪天说了什么这类"为什么"归到对应的 decisions/pitfalls 文件,用双链指过去)
3. **不要靠"在同一段话里抠字数"来减重**——这条路径边际收益很低(2026-07-06 实测过,两次同段落压缩合计只减了 5.7%);真正有效的是把整块"任务特定"的内容搬出无条件加载的范围

# 已经做过的拆分(参考先例)

- `RESOLVER.md`:该往哪个记忆分类写、写不写——只在判断"要不要写记忆"时读
- `WORKBENCH.md`:工作台每日生成的完整规范——只在处理工作台任务时读
- `GC.md`:Memory GC 步骤 + notes 复核节奏——只在执行 GC 或处理复核到期时读

2026-07-06 这次实测:CLAUDE.md 从 10,592 字符降到 6,073 字符(降了 42.7%),换算下来大概相当于用户观察到的"~7% context window 占用"降到 4% 左右——同时三份被搬出去的内容一个字都没删,需要时随时能查,操作性完全没打折扣。

# 后续维护提醒

以后每次给 CLAUDE.md 加新规则前,先想一遍这条规则是不是"几乎所有会话都用得上"——不是的话直接建新文件,不要图省事先塞进 CLAUDE.md 再说。
